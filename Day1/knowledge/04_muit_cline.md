# MUITとClineが語る、AIコーディングエージェント組織展開の現実

**イベント:** AI Engineering Summit Tokyo 2026 / Day 1  
**発表者:** 三菱UFJインフォメーションテクノロジー（MUIT）× Cline

---

## 概要

AIコーディングエージェントを企業組織に展開する際の現実的な課題と解決策を、Cline（製品側）とMUIT（導入側）の両視点から語ったセッション。エージェント型開発に必要な「柔軟性」と「標準化」を、ツール選択にどう反映させるかが主テーマ。

---

## 主要なポイント

- プロジェクトで1つのモデルや1つのサービスを決めることは**最新のキャッチアップを妨げる**
- エージェント型開発には**モデル・ツール両方への柔軟な対応能力**が求められる
- MUITの学習ループ: **素早い実験 → 標準化 → 組織化**というサイクルが有効
- **Cline = 柔軟性（実験フェーズ）**、**Claude Code = 標準化（展開フェーズ）**という使い分けが実践されている
- コーディングエージェントの企業導入で最も難しいのは「**AIを取り込んだプロセス変革**」と「**マインドセットの変容**」

---

## 詳細内容

### Cline（AIコーディングエージェント）の特徴

**The agentic coding platform with choice（選択の自由があるプラットフォーム）**  
Clineの最大の特徴は、使用するモデルを自由に選べることにある。

- **フロンティアモデルでの性能** — Claude、GPT、Geminiなど主要なフロンティアモデルに対応し、Terminal Benchなど各種ベンチマークでのスコアが高い
- **オープンウェイトモデルにも対応** — クローズドなAPIに依存せず、ローカルモデルも活用可能
- **Open-source Cline SDK** — パッケージ構造がオープンで、独自のエージェントを構築するためのランタイムとして活用可能
- **For Developers** — 個人の開発生産性向上に最適化
- **For Teams** — チームでの標準化・共有機能を提供
- **npx skills add** — スキル（Agent Skills）を追加するコマンドで、必要な機能をモジュール式に拡張できる

### MUITが直面したエンタープライズAIDD（AI-Driven Development）の課題

**Why Enterprise AIDD Is Hard**  
- 適切なものを構築できているかどうかわからない
- AIを取り込んだプロセスを導入する変化が難しい
- マインドを変える必要がある

**Challenge 1: Methodology Gap（方法論のギャップ）**  
既存の開発プロセスとAIを活用した開発プロセスの間に大きなギャップがある。特に企業の場合、承認フローや品質基準が固定されており、AIを組み込んだ際にどうプロセスを再設計するかが課題になる。

**Challenge 2: Skill Gap（スキルギャップ）**  
モデルもツールも急速に進化する中、エンジニアがキャッチアップし続けることが困難。また、「AIエンジニアとしてのスキル」は従来のエンジニアスキルとは異なる部分があり、組織的な育成が必要。

### MUITの学習ループ

エンタープライズでのAI活用を組織に根付かせるために、以下の3段階のループを回している。

1. **プロジェクト実行（素早い実験）** — 実際のプロジェクトでAIコーディングエージェントを試し、知見を蓄積する
2. **標準化（モデル、エージェント）** — 実験から得られた有効なパターンをモデル・エージェントとして標準化する
3. **組織化（CoE、FDEチーム）** — 標準化された手法を組織に展開するCenter of Excellence（CoE）やFDEチームを組成する

### ClineとClaude Codeの使い分け戦略

**セキュリティ・組織的制約（閉鎖環境）を考慮した上で**、フェーズによって使い分けを行っている。

| フェーズ | ツール | 理由 |
|---------|--------|------|
| 素早い実験 | Cline | 使用するモデルの柔軟性が高く、さまざまなモデルを試せる |
| 標準化・組織化 | Claude Code | Agent Skills等の標準化機能が充実しており、オンボーディングに強い |

**Clineの強み**
- 使用モデルの柔軟性
- カスタマイゼーション性
- オープンソース

**Claude Codeの強み**
- Agent Skillsなどの標準化されたエコシステム
- オンボーディングのしやすさ
- 組織展開への適性

### パネルディスカッションで出た知見

- 最新モデルのキャッチアップは重要だが、それ自体が目的化してはいけない
- AIの出力が正しいことを証明することが難しい（検証フローの整備が必要）
- アーリーアダプターであることがAIエンジニアとしての競争優位になる
- コーディングエージェントを企業で使う本当の難しさは技術ではなく、**変化のマネジメント**と**マインドセットの転換**

## 社内への示唆・活用ポイント

- **ツールを1つに固定しない** — モデルもツールも進化が早い。特定のツール・モデルにロックインせず、定期的に見直す体制を整える
- **実験フェーズと展開フェーズでツールを分けることを検討** — 柔軟性が必要な試験段階（Cline等）と、組織展開が必要な標準化段階（Claude Code等）で最適なツールは異なる
- **学習ループを組織設計に組み込む** — 実験 → 標準化 → 組織化のサイクルを意識的に設計しないと、実験が散発的に終わり組織全体に知見が広がらない
- **AI出力の検証フローを事前に設計する** — AIが生成したコードやドキュメントが「正しい」かどうかを検証する仕組みを、AI導入と同時に整備する必要がある
- **CoEまたはFDEチームの設置を検討** — AIコーディングの知見を組織横断で蓄積・展開する専門チームが、大規模展開には有効
