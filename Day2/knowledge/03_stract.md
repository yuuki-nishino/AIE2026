## 概要

20人というコンパクトなチームで100億円規模のコマースアプリ事業を運営するSTRACTが、AIを使った「E2E AI-Native」な開発・運用体制のリアルを紹介したセッション。重要なのは完全自律化よりも「構造で制御する」という考え方であり、人間とAIの役割分担を明確に設計することで、小規模チームでも大規模事業を動かせることが示された。

---

## 主要なポイント

- 「**構造で制御する**」ことがE2E AI-Native開発の本質。構造で制御できないものだけプロンプトやHITLで対応する
- **失敗を構造で消す** — 再現可能な失敗は、システム設計で根絶する
- **LLMの回答を信用しない** — エビデンス（画面録画等）を残させることが重要
- AIは**仮説を出す役**、人間は**評価・承認する役**という分業が現実解
- **Macハードウェアをオフィスに置いて24時間365日開発**できる環境を整備する

## 詳細内容

### システムアーキテクチャ

```
INTEGRATION      ORCHESTRATOR    WORKER
GitHub ─────webhook──→            ─dispatch→  git worktree #1 (Claude Agent SDK)
Linear ──────────────→            ─dispatch→  git worktree #2 (Claude Agent SDK)
                              ←── think/ask/done ───────────────────────
```

**設計の3原則：**
- **並列で干渉しない** — 1 Issue = 1 git worktreeを割り当て、複数タスクが並列で干渉せずに動く
- **失敗しても再開** — worktreeが失敗しても途中から再開できる設計
- **1 Issue = 1 worktree** — タスクの単位を明確に区切ることで管理性を高める

### 信頼性を確保するための4原則

**1. 人の承認を入れる**  
書き込み操作・重要な変更は必ず人間の承認ステップを経る。AIに任せる範囲を「読む・考える・提案する」に限定し、「実行」には人間の承認を必ず入れる設計。

**2. Deterministicなフォールバックを作る**  
AIが判断できない・信頼できないケースに備え、決定論的なフォールバック処理を用意しておく。「AIが動かなくても業務が止まらない」バックアップを常に持つ。

**3. LLMの回答を信用しない（エビデンスを残させる）**  
AIが「完了した」と言っても、本当に動作したかどうかは別途確認する。具体的には**画面録画**を取得させてエビデンスを残させる。LLMの自己申告ではなく、実際の動作記録で検証する。

**4. 失敗を構造で消す**  
再発する失敗はシステムの構造的欠陥。プロンプト修正やHITLで個別対処するのではなく、失敗が起きない構造に設計し直す。構造で消せないものだけをプロンプトやHITLで対応する。

### 自社ハーネスを作るメリット

既製品のエージェント基盤を使うのではなく、独自のハーネス（実行環境）を構築することには以下のメリットがある。

- **Mac環境で動作する** — Macハードウェアをオフィスに物理設置し、24時間365日稼働させることができる
- **カスタマイズ性が高い** — 自社の業務要件に合わせた最適化が可能
- **コスト最適化** — クラウドのコンピューティングコストを抑え、必要なスペックを確保できる

### Macハードウェアを使った24/365開発

STRACTの独自アプローチとして注目すべき点は、**Macのハードウェアをオフィスに設置してWorkerとして使う**という方法。AWSやGCPの仮想インスタンスではなく、物理のMacを使うことで:
- Xcodeが必要なiOS開発などMac固有の環境が必要なタスクにも対応できる
- 24時間365日のバックグラウンド開発が可能になる

### 全業務のAI置き換えという目標

「全業務をAIで置き換えよう」という目標を掲げている。これは「人間の仕事を奪う」という意味ではなく、**ルーティン業務・定型業務をAIに移管し、人間はより創造的・判断的な業務に集中する**という意味。

実際に動くかどうかを確かめ続けることが前提であり、動作確認なしに全自動化するという意味ではない。

## 社内への示唆・活用ポイント

- **「AIを信じる」前に「エビデンスで確かめる」習慣を徹底する** — LLMが「完了した」と報告しても、その結果を人間・システムが独立して検証できる仕組みを作る。画面録画・ログ・成果物の差分比較など、確認手段を事前に設計しておく
- **「構造で制御」を設計の出発点にする** — プロンプトで振る舞いを制御しようとする前に、構造（ワークフロー・権限・フォールバック）で制御できないかを先に考える
- **承認フローは「文化」ではなく「システム」で実装する** — 「重要な操作は承認を得て」というルールではなく、技術的に承認なしでは実行できない設計にする
- **1 Issue = 1 worktreeの分離単位は参考になる** — 複数のAIタスクを並列で走らせる場合、互いに干渉しない単位に分離することが、信頼性向上の鍵。git worktreeはその実装例として有効
- **フォールバックを先に作る** — AIが動いて初めてフォールバックを考えるのではなく、AIなしでも動く通常フローを先に確保した上でAIを導入する順序が安全
